Фаза 3 — Архитектура для масштаба (1000 юзеров)
3.1. REST API вместо только CLI

CLI не масштабируется на 1000 пользователей. Добавить HTTP API:

POST /api/users — регистрация
GET /api/users — список
POST /api/schedules — добавить расписание
GET /api/schedules/:user_id — расписания пользователя
DELETE /api/users/:id, DELETE /api/schedules/:id
CLI оставить как обёртку над API (или как отдельный admin-инструмент).

3.2. Асинхронная отправка писем

Сейчас отправка синхронная — 1 письмо за раз. При 1000 юзерах в один слот (например, Пн 8:00) отправка займёт 5-8 минут.

Решение: worker pool из горутин (5-10 воркеров). Без внешних очередей — для 1000 юзеров это overkill.

3.3. Connection pooling

Настроить SetMaxOpenConns(25), SetMaxIdleConns(5), SetConnMaxLifetime(5m) в db.go.

3.4. Таймзоны

Добавить поле timezone в таблицу users. Шедулер проверяет время с учётом TZ каждого пользователя.

3.5. Rate limiting для Yandex SMTP

Yandex ограничивает ~30 писем/минуту. Добавить rate limiter, чтобы не получить бан.
