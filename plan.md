# План доведения проекта до релиза

> **Цель:** рабочий продукт на ~1000 пользователей/месяц.
> **Таймзона:** Moscow (UTC+3). **Язык:** русский. **SMTP:** Yandex. **Авторизация:** логин + пароль.
> **AI-персонализация:** DeepSeek API. **Защита от ботов:** rate limit + CSRF.
> **UI:** server-side rendering (Go templates + Pico.css).

---

## Текущее состояние проекта

- Go CLI-приложение (без веб-интерфейса), PostgreSQL, Yandex SMTP
- Фазы 1-4 выполнены (чистка + критические баги + рефакторинг + валидация и обработка ошибок)
- Нет тестов, нет веб-интерфейса, нет авторизации, нет AI-генерации
- Все тренировки и питание — захардкоженные шаблоны (4 варианта тренировок, 3 варианта питания)
- Архитектура: пакеты `internal/*`, Store без глобального `db`, `log/slog`, `html/template`
- Валидация: пакет `internal/validation`, проверка email/числовых полей, дедупликация, graceful scheduler

## Фаза 6. Авторизация (логин + пароль)

### 6.1 Хеширование паролей
`golang.org/x/crypto/bcrypt`, cost=12. Пакет `internal/auth`. Хранить в существующем поле `password_hash`.

### 6.2 Регистрация
При создании пользователя — пароль мин. 6 символов, хеш bcrypt.

### 6.3 Сессии
Session cookie (HttpOnly, Secure, SameSite=Strict) или JWT. Middleware: нет токена → 401, невалидный → 401, валидный → `userID` в context. Пользователь может редактировать **только свои** данные.

## Фаза 7. Web API + интерфейс + антиспам

### 7.1 HTTP-сервер
`net/http` + `http.ServeMux`. Обновить Go до 1.22+ для маршрутизации с методами. Новая команда `serve` — HTTP + scheduler одновременно.

Go `html/template` с layout-наследованием. Стилизация: Pico.css через CDN. Всё на русском.

### 7.4 Отписка из письма
Ссылка в каждом email. JWT с userID + scheduleID, срок 30 дней. Клик → `is_active = false`.

### 7.5 Защита от спама и ботов

**Rate limiting по IP:** register 3/час, login 10/15мин, остальные 60/мин. Реализация через `golang.org/x/time/rate` + map по IP с TTL. Очистка просроченных записей каждые 10 минут.

**CSRF-токены:** `crypto/rand` токен при загрузке страницы → cookie (HttpOnly, SameSite=Strict) + hidden field в форме. При POST — проверка совпадения.

**Honeypot-поля:** скрытое поле `website` (display:none) в формах регистрации и логина. Заполнено → бот, тихий 200 OK.

**Блокировка логина:** 3 неудачи → 30сек задержка, 5 → блок 15мин. "Неверный email или пароль" — одинаковое сообщение всегда.

**Security headers:** X-Content-Type-Options, X-Frame-Options, CSP, Referrer-Policy.

### 7.6 Health check
`/health` — ping БД, 200 "ok" / 503 "db down".

## Фаза 8. AI-персонализация через DeepSeek

Сейчас тренировки и питание — захардкоженные шаблоны. 4 варианта тренировок, 3 варианта питания. Пользователь с одинаковой целью получает идентичные письма каждый день.

### 8.1 Клиент DeepSeek API
Пакет `internal/ai/`. OpenAI-совместимый формат. Конфиг: `DEEPSEEK_API_KEY`, `DEEPSEEK_MODEL` (default: `deepseek-chat`). Таймаут 30 секунд. Стоимость: ~$0.5-1/месяц при 1000 пользователях.

### 8.2 Промпт-система (3 промпта)

**Тренировка:** учитывает пол, возраст, рост, вес, цель, уровень активности, день недели, время суток. Ответ JSON: title, exercises (4-6 с подходами/повторениями), duration. НЕ повторять группы мышц подряд, разнообразие упражнений, учёт времени суток (утром бодрящие, вечером спокойнее).

**Питание:** конкретные блюда с граммовками, доступные в РФ. JSON: breakfast, lunch, dinner, snacks, calories, protein, fat, carbs, water_ml. Расчёт БЖУ и ккал на основе данных пользователя.

**Мотивация:** 2-3 предложения, юмор и подколки (как друг, который тащит в зал), эмодзи. Ключевая фишка продукта — каждое письмо уникальное.

### 8.3 Чередование мышечных групп
Новая таблица `workout_history` (user_id, date, muscle_group). Передавать в промпт информацию о предыдущих тренировках. AI учитывает и не повторяет. Цикл замкнутый: за несколько дней прорабатываются все группы мышц.

### 8.4 Retry и fallback
3 попытки: 2с → 5с → 10с. Между пользователями 300мс пауза. Полный провал → письмо-заглушка с извинением (не молчать, пользователь должен знать что подписка работает).

### 8.5 Парсинг ответа
Извлечение JSON из текста (может быть обёрнут в markdown-блок). Валидация: есть exercises, duration, калории. Ошибка парсинга = провал попытки → retry.

### 8.6 Обновить шаблон письма
Удалить захардкоженные функции генерации тренировок. Новый шаблон для AI-контента.

### ✅ Верификация фазы 8
- Два письма одному пользователю в разные дни — разные тренировки
- Мышечные группы чередуются (нет двух дней ног подряд)
- DeepSeek недоступен → письмо-заглушка после 3 retry
- Питание с реалистичными продуктами, граммовками и БЖУ

## Фаза 9. Подготовка к нагрузке (1000 пользователей)

### 9.1 Таблица `email_logs`
Поля: user_id, schedule_id, sent_at, status (sent/failed/retrying), error_message. Дедупликация: перед отправкой проверить, не было ли уже письма сегодня по этому расписанию.

### 9.2 Rate limiting SMTP
Yandex ≈500 писем/день. 1000 пользователей × 3 дня/неделю ≈ 430/день (впритык). Пауза 200мс между письмами. Ошибка 429 → exponential backoff. Логировать дневной счётчик, предупреждать при приближении к лимиту.

### 9.3 Retry SMTP
3 попытки: 1с → 5с → 15с. Записывать результат в `email_logs`.

### 9.4 Заменить gomail.v2
`gopkg.in/gomail.v2` не обновлялся с 2016 года → `github.com/wneessen/go-mail` (активная поддержка, TLS 1.3).

### ✅ Верификация фазы 9
- `email_logs` заполняется при каждой отправке
- Дедупликация работает
- Retry работает
